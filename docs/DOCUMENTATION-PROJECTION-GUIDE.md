# Documentation Projection Guide for Extension Authors

**Epic:** EXTENSION-SDK-DOCUMENTATION-HOOKS-1
**Status:** Ratified
**Date:** 2026-07-22

---

## 1. What This Is

The Librarian documentation projection system generates human-readable documentation from canonical governance state. As an extension author, you do **not** implement your own generator. You use the Librarian's `librarian_documentation_generate` MCP tool to produce documentation for your extension.

The SDK provides templates with the required marker structure so that generated documentation can be regenerated without destroying your human-authored content.

## 2. Architecture

```
Librarian Canonical State
         |
         | read-only
         ▼
librarian_documentation_generate (Librarian MCP tool)
         |
         ├── Generated Markdown sections
         │      └── bounded by LIBRARIAN-GENERATED-SECTION markers
         │
         └── Projection Receipt
                ├── source_hash (SHA-256 of canonical state)
                ├── generated_at (timestamp)
                └── generator_version
```

**Your extension provides:**
- The file with marker structure (template provided by SDK)
- Human-authored architecture notes outside markers

**Librarian provides:**
- The generated content within markers
- The projection receipt
- Drift detection against canonical state

## 3. Supported Projection Types

| Projection Type | Tool Argument | Output Content |
|----------------|---------------|----------------|
| extension_status | `type: "extension_status"` | Extension identity, lifecycle state, capabilities, contract info |
| capability_map | `type: "capability_map"` | All active capabilities across all extensions |
| project_status | `type: "project_status"` | Project-wide extension ecosystem summary |

For extension documentation, use `extension_status`.

## 4. Marker Convention

Generated files use two types of markers:

### Projection Receipt

Embedded at the top of the file as an HTML comment:

```markdown
<!-- LIBRARIAN-PROJECTION-RECEIPT
{
  "projection_id": "docproj-20260722-...",
  "source_hash": "sha256:...",
  "generated_at": "2026-07-22T00:00:00Z",
  "generator_version": "1.0.0",
  "projection_type": "extension_status",
  "status": "current"
}
-->
```

This receipt is the evidence that connects the document to canonical state.

### Generated Section

Content that will be replaced on regeneration:

```markdown
<!-- LIBRARIAN-GENERATED-SECTION -->
<!-- Generated by librarian_documentation_generate -->

... content that will be overwritten on next generation ...

<!-- END LIBRARIAN-GENERATED-SECTION -->
```

Content outside these markers is preserved across regeneration cycles.

## 5. Workflow

### Initial Setup

After creating your extension with the SDK, the template already includes the correct marker structure. To generate your extension status document:

```bash
librarian_documentation_generate \
  --type extension_status \
  --extension-id your-extension-id \
  --output-path docs/EXTENSION-STATUS.md
```

### Regeneration

When your extension's lifecycle state changes (e.g., transitions from REGISTERED to ACTIVE), regenerate the status document:

```bash
librarian_documentation_generate \
  --type extension_status \
  --extension-id your-extension-id \
  --output-path docs/EXTENSION-STATUS.md
```

The generator will:
- Replace the content between `LIBRARIAN-GENERATED-SECTION` markers
- Update the embedded `LIBRARIAN-PROJECTION-RECEIPT`
- Preserve all human-authored content outside the markers

### Drift Detection

To check whether a generated document reflects current state:

```bash
librarian_documentation_drift_check \
  --file-path docs/EXTENSION-STATUS.md
```

Returns:
- `PASS` — document is current
- `OBSERVATION` — state has changed since generation
- `ERROR` — receipt missing or malformed

## 6. What NOT To Do

| Don't | Why |
|-------|-----|
| Implement your own projection generator | The generator is a Librarian-core capability. Extensions declare capabilities; Librarian projects them. |
| Modify content inside `LIBRARIAN-GENERATED-SECTION` markers | Your edits will be overwritten on next regeneration. |
| Remove the `LIBRARIAN-PROJECTION-RECEIPT` | The generator requires a valid receipt to regenerate. Missing receipt → ERROR. |
| Treat generated documentation as governance authority | Documentation is a projection of state, not the state itself. Always query canonical state for decisions. |
| Generate a file that has no markers | The generator refuses to run on files without `LIBRARIAN-GENERATED-SECTION` markers. |

## 7. Template Files

The SDK provides the following templates with documentation projection hooks:

| File | Markers | Human Content |
|------|---------|---------------|
| `docs/EXTENSION-STATUS.md` | Full receipt + generated section | Architecture notes, regeneration instructions |
| `docs/PROJECT-IDENTITY.md` | Generated section for current state | Purpose, non-goals, governance boundaries |
| `README.md` | Generated section for status summary | Quick start, implementation plan, compliance checklist |

## 8. Verification

After setting up documentation projection for your extension:

1. **Generate initial document:**
   ```bash
   librarian_documentation_generate --type extension_status \
     --extension-id <id> --output-path docs/EXTENSION-STATUS.md
   ```

2. **Verify drift check passes:**
   ```bash
   librarian_documentation_drift_check --file-path docs/EXTENSION-STATUS.md
   # Expected: PASS
   ```

3. **Transition extension state** (e.g., activate or suspend)

4. **Regenerate and verify drift detected:**
   ```bash
   librarian_documentation_drift_check --file-path docs/EXTENSION-STATUS.md
   # Expected: OBSERVATION (state changed)
   ```

5. **Regenerate to restore currency:**
   ```bash
   librarian_documentation_generate --type extension_status \
     --extension-id <id> --output-path docs/EXTENSION-STATUS.md
   ```
